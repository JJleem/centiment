결정 1 — 플랫폼 선택 제거
왜 iOS / Android 선택을 없앴나?
플랫폼을 따로 분석하면 "iOS 유저는 게임플레이를 좋아한다"는 맥락 없는 숫자만 남음. 비교가 있어야 "iOS 긍정률이 Android보다 14%p 높다"는 인사이트가 생김.

분석 비용은 두 배지만 병렬 실행(Promise.all)으로 시간은 단일 플랫폼과 동일.
❌ 기각된 안: 플랫폼 선택 유지 + 별도 비교 페이지 → 분석을 두 번 실행해야 비교 가능, UX 마찰 큼
결정 2 — 수집 건수 선택 (100 / 200)
왜 건수를 사용자가 선택하나?
분석 시간과 비용은 건수에 비례. 100건으로 충분한 인사이트를 얻을 수 있지만, 장기 트렌드나 정밀 분석이 필요한 경우 200건이 필요.

시스템이 자동 결정하는 것보다 트레이드오프를 사용자에게 투명하게 노출하는 것이 내부 툴에 맞는 설계.
⚠️ 200건 선택 시 경고 배너 표시 → 사용자가 인지하고 선택하도록
결정 3 — 게임 간 비교 UI를 랜딩 하단에 배치
왜 별도 페이지가 아닌 랜딩에?
/cross 페이지는 직접 URL을 알아야 접근 가능. 경영/마케팅 레이어 사용자는 URL을 외우지 않음. 랜딩에 두 게임 픽커를 두면 분석 히스토리와 자연스럽게 연결되고 발견성이 높아짐.
비교 버튼이 언제 활성화되나?
두 게임을 모두 선택하고 서로 다른 게임일 때만 활성. 같은 게임 선택 시 "서로 다른 게임을 선택해 주세요" 안내 표시. 비교 불가 상태에서 클릭 기회 원천 차단.
결정 4 — 히스토리: 게임별 그룹핑 (iOS+Android 하나로)
왜 iOS와 Android를 하나의 히스토리 항목으로 묶었나?
기존에는 "Pizza Ready iOS", "Pizza Ready Android" 두 줄이었음. 사용자 입장에서 "Pizza Ready 분석 결과"가 의미 단위. 두 플랫폼이 분석되어 있으면 iOS·Android 아이콘을 함께 표시하고 클릭 시 /result?game=pizza-ready로 이동. 한쪽만 분석된 경우에도 아이콘 색상으로 구분.
❌ 기각된 안: app_id+platform 단위 분리 히스토리 → 같은 게임이 두 줄로 표시되어 혼란

Centiment — Result Page
결과 페이지 — /result?game=…
URL 파라미터 설계
왜 ?app_id=&platform= 대신 ?game=?
iOS와 Android는 서로 다른 app_id를 사용 (iOS: 숫자 ID, Android: 패키지명). 사용자에게 의미 있는 단위는 "게임"이지 "앱 ID"가 아님. game 파라미터 → SUPERCENT_GAMES preset에서 두 app_id를 모두 파생해 병렬 조회.
왜 Server Component?
분석 결과는 정적. Supabase 조회를 서버에서 직접 실행해 클라이언트로 데이터를 전송하지 않음. 단, 키워드 드릴다운은 인터랙션이 필요해 KeywordDrilldown.tsx만 Client Component로 분리.
섹션 배치 순서와 이유
① AI 인사이트
2열
iOS vs Android 각각의 3문장 요약. 비교 맥락의 시작점.
② 이슈 목록
버그 있을 때만
버그는 즉각 행동이 필요한 정보. 조건부 렌더링으로 없으면 카드 자체 숨김.
③ 감성 비교
2열 + 차이 표시
긍정률 차이를 N%p로 명시. 어느 플랫폼 유저가 더 만족하는지 즉시 파악.
④ 카테고리 비교
이중 바차트
iOS/Android 같은 카테고리를 이중 바로. 플랫폼별 불만 영역 차이 시각화.
⑤ 버전 트렌드
≥2 버전 시
업데이트 후 감성 변화 추적. iOS·Android 각각 별도 차트.
⑥ 키워드
+ 리뷰 통합
클릭 시 해당 키워드 리뷰만 필터. 공통 키워드 볼드. 리뷰는 날짜순 병합.
키워드 드릴다운 아키텍처
왜 URL param이 아닌 로컬 state?
키워드 필터는 일시적 탐색 행위. URL에 올리면 뒤로 가기, 공유 시 필터 유지 등 불필요한 복잡도 발생. 같은 페이지 내 탐색이므로 useState로 충분.
왜 공통 키워드를 강조하나?
iOS와 Android 유저 모두 같은 키워드를 사용한다면 플랫폼에 무관한 게임 자체의 특성. 공통 키워드 = 가장 보편적인 유저 경험. 볼드 처리로 즉시 식별.

Centiment — AI Engine
AI 분석 엔진 설계
모델 역할 분리 — 왜 Haiku와 Sonnet을 나눴나
🟠 Haiku = 분류 전담
리뷰 분류는 패턴 인식 작업. "이 리뷰가 버그 관련인가, 게임플레이 관련인가"는 Haiku의 속도·가격으로 충분히 정확. 20건 배치로 처리해 단건 대비 비용 1/20 수준.

Sonnet을 쓰면 정확도가 소폭 올라갈 수 있지만 비용이 5~10배 증가.
🟣 Sonnet = 요약 + 이슈 전담
"버그 리뷰들에서 공통 이슈를 추출해 한국어 3문장 요약 + 구체적 이슈 목록을 만들어라"는 추론과 합성 작업. Haiku는 뉘앙스를 놓침.

Sonnet은 플랫폼당 단 1회만 호출 — 아무리 리뷰가 많아도 Sonnet 비용은 고정.
핵심 최적화 — Sonnet 1회 호출로 요약 + 이슈 목록 동시 생성
왜 이슈 목록을 별도 API 호출로 만들지 않았나?
버그 이슈 목록을 만들기 위해 Sonnet을 한 번 더 호출하면 플랫폼당 Sonnet 2회 → iOS+Android = 4회. 비용과 레이턴시가 4배.
Sonnet 요약 프롬프트에 버그 원문(최대 20건)을 추가 입력으로 넣고 JSON { summary, issues }를 단일 출력으로 받으면 동일 비용으로 두 가지 산출물이 나옴.
// Sonnet 입력
{ stats: 감성통계, topKeywords, categoryCounts, bugContents: string[] }

// Sonnet 출력
{ "summary": "3문장 한국어 요약", "issues": ["광고 스킵 불가", "결제 오류"] }
배치 처리 — 왜 20건 단위인가
1건씩 호출하지 않는 이유
100건 분석 시 1건씩 = Haiku 100회 호출. 20건 배치 = 5회. API 오버헤드, 레이턴시, 비용 모두 20배 차이. 배치 내 리뷰 간 맥락 유지도 분류 일관성에 도움.
20건 이상으로 늘리지 않는 이유
배치가 너무 크면 Haiku의 응답 JSON이 길어져 파싱 오류 발생률이 높아짐. 20건은 max_tokens 4096 내에서 안정적으로 처리되는 검증된 크기.

Centiment — Incremental Analysis
증분 분석 아키텍처
Before vs After

❌ 기존 — 전체 삭제 후 재분석

1. 전체 리뷰 조회 (app_id + platform)
2. review_analysis 전부 DELETE
3. 전체 리뷰 Haiku 재분류 (100건 = 5배치)
4. Sonnet 요약
5. 전체 INSERT
→ 5건 신규 리뷰를 위해 100건 전체 재처리

✅ 현재 — 신규 리뷰만 분석

1. 기존 분석된 review_id 목록 조회
2. 전체 리뷰 조회
3. 미분석 리뷰만 필터 (review_id NOT IN 기존)
4. 신규 리뷰만 Haiku 분류 (5건 = 1배치)
5. INSERT (신규 행만)
6. Sonnet 전체 집계로 요약 재생성 → UPDATE
→ 5건만 처리. Haiku 비용 1/20, 시간 대폭 감소
review_id FK가 핵심인 이유
FK 없이는 증분이 불가능했던 이유
기존 review_analysis는 reviews와 별도 테이블로 인덱스 기반 페어링 구조. "어떤 리뷰가 분석됐는지" 알 방법이 없음. review_id를 FK로 추가한 뒤 WHERE id NOT IN (SELECT review_id FROM review_analysis) 패턴으로 미분석 리뷰만 추출 가능.
force 모드는 언제 쓰나
프롬프트를 수정했거나, 분류 기준이 바뀌어서 전체를 다시 분류해야 할 때. ReanalyzeButton에서 "전체 재분석" 작은 버튼으로 접근 가능. force=true 시 DELETE + 전체 재분석.
외부 API 호출 — 현재 한계와 다음 단계
현재: DB 레벨 증분 (구현 완료)
App Store / Play Store는 여전히 매번 전체 호출. 단, upsert + ignoreDuplicates로 DB에는 신규 리뷰만 추가. 분석은 review_id FK 기반으로 신규만 처리. Claude 비용 기준으로는 증분 달성.
다음 단계: 스크래퍼 레벨 최적화
DB에서 최신 review_date 조회 → 스크래퍼에서 그 날짜 이후 리뷰만 처리 후 insert 시도 → 이미 있는 날짜의 리뷰가 나오면 early exit. 외부 API 호출 수 자체를 줄이는 방향.

왜 iOS+Android를 통합 집계하나?
타이틀 비교의 목적은 "Pizza Ready가 Snake Clash보다 유저 만족도가 높은가". 이 질문에 플랫폼 차이는 노이즈. iOS+Android를 합산해 게임 전체의 대표 수치를 만드는 것이 비교의 공정한 기준.
❌ 기각: 플랫폼별 4열 비교 → 인지 부하 과다, 핵심 메시지 희석
공통 키워드 볼드 처리의 의미
두 게임에서 공통으로 등장하는 키워드 = 특정 타이틀 특성이 아닌 장르 공통 특성. "두 게임 모두 monetization 키워드가 높다"면 장르 전체의 개선 기회. 볼드로 즉시 식별.
핵심 설계 결정 요약
결정 채택한 방향 핵심 이유

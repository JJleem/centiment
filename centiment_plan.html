왜 단방향인가
수집·분석·표시를 파이프라인으로 고정하면 각 단계를 독립적으로 교체하고 테스트할 수 있다. AI 분석 결과가 수집 로직에 영향을 주지 않기 때문에 어느 단계에서 문제가 생겼는지 즉시 격리할 수 있다.

Centiment — Landing Page /
랜딩 페이지
iOS · Android 동시 분석
게임만 선택하면 두 플랫폼을 한번에 분석한다
분석 버튼 클릭 시 iOS fetch → iOS analyze, Android fetch → Android analyze를 Promise.all로 병렬 실행. 총 소요 시간은 단일 플랫폼과 동일하다.
왜
플랫폼 하나만 보면 숫자는 있지만 맥락이 없다. "iOS 긍정률 72%"는 Android와 비교해야 높은지 낮은지 알 수 있다. 비교가 이 대시보드의 핵심 가치.
수집 건수 선택 (100 / 200건)
사용자가 직접 분석 규모를 결정한다
100건은 빠른 인사이트, 200건은 장기 트렌드 파악에 적합. 200건 선택 시 "분석 시간이 더 걸릴 수 있습니다" 경고 표시.
왜
분석 시간과 비용은 건수에 비례한다. 내부 툴에서는 트레이드오프를 사용자에게 투명하게 노출하고 판단을 위임하는 것이 맞다.
게임 간 비교 진입 UI
랜딩 하단에서 두 게임을 선택해 /cross로 이동
첫 번째 게임(indigo)과 두 번째 게임(violet)을 각각 선택. 두 게임이 모두 선택되고 서로 다를 때만 비교 버튼이 활성화된다.
왜 랜딩에 두었나
/cross URL을 직접 타이핑해서 들어오는 사용자는 없다. 분석 히스토리 바로 위에 두어 "분석 결과가 쌓였을 때 자연스럽게 비교"하는 흐름으로 연결.
분석 히스토리
게임 단위로 그룹핑, 클릭 시 /result로 이동
iOS와 Android가 분석돼 있으면 하나의 히스토리 항목으로 표시. 감성 미니 바로 긍정/부정/중립 비율을 한눈에 확인. 플랫폼 아이콘 색상으로 어느 쪽이 분석됐는지 표시.
왜 게임 단위로 묶나
사용자에게 의미 있는 단위는 "Pizza Ready 분석 결과"이지, "iOS app_id 6450917563 분석 결과"가 아니다.

Centiment — Result Page /result
결과 페이지 — /result?game=…
URL 파라미터: ?game=pizza-ready
게임 ID 하나로 iOS·Android 두 개의 app_id를 파생
iOS는 숫자 ID(6450917563), Android는 패키지명(io.supercent.pizzaidle)으로 서로 다른 키를 사용한다. game 파라미터를 받으면 SUPERCENT_GAMES 프리셋에서 두 값을 꺼내 병렬로 DB 조회.
왜
app_id를 URL에 노출하면 iOS/Android 중 하나만 보는 구조가 되어버린다. 게임이 의미 단위이므로 URL도 게임 기준.
재분석 버튼 — 두 가지 모드
"새 리뷰 분석"과 "전체 재분석"을 분리
새 리뷰 분석(기본): 아직 분석되지 않은 리뷰만 처리. iOS+Android 동시 실행.

전체 재분석: 기존 분석 결과를 모두 지우고 처음부터 다시. 프롬프트를 수정했을 때 사용.
섹션 구성과 배치 이유
①
AI 인사이트 — iOS / Android 2열
Sonnet이 생성한 3문장 요약. 각 플랫폼 유저가 무엇을 좋아하고 무엇에 불만인지를 다른 언어로 읽을 수 있다. 숫자 차트 전에 텍스트 맥락을 먼저 제공.
②
이슈 목록 — 버그 리뷰가 있을 때만 표시
버그 카테고리 리뷰에서 추출한 구체적 이슈 목록. "광고 스킵 불가", "결제 후 아이템 미지급" 등. 즉각 행동이 필요한 정보이므로 상단 배치. 버그 리뷰가 없으면 카드 자체가 나타나지 않는다.
③
감성 비교 — iOS / Android 2열 + 긍정률 차이 표시
긍정/부정/중립 바차트를 나란히. 두 플랫폼의 긍정률 차이를 "%p" 단위로 명시해 한 줄로 결론을 읽을 수 있다.
④
카테고리 비교 — 이중 바차트
게임플레이·버그·결제 등 7개 카테고리별로 iOS 바와 Android 바를 한 쌍으로 표시. 어떤 카테고리에서 플랫폼 차이가 큰지 시각적으로 파악.
⑤
버전별 감성 트렌드 — 2개 이상 버전이 있을 때만 표시
업데이트마다 감성이 어떻게 바뀌었는지 스택 바차트로 표시. iOS·Android 각각 별도 차트. 버전 정보가 없거나 단일 버전이면 차트가 나타나지 않는다.
⑥
키워드 비교 + 드릴다운 + 리뷰 목록
키워드 클릭 시 해당 키워드를 포함한 리뷰만 필터링. 다시 클릭하면 해제. iOS·Android 공통 키워드는 진하게 강조. 리뷰 목록은 iOS+Android를 날짜순으로 합쳐 표시하며 플랫폼·감성·카테고리 필터 제공.

Centiment — AI Engine
AI 분석 엔진
모델 역할 분리
🟠 Haiku — 분류 전담
리뷰 20건을 한 배치로 묶어 호출. 각 리뷰마다 sentiment(긍정/부정/중립), category(버그/게임플레이 등 7종), keywords(최대 3개)를 JSON 배열로 반환.
왜 Haiku인가
분류는 패턴 인식 작업이라 Haiku 속도·가격으로 충분히 정확하다. 20건 배치 처리로 단건 대비 API 호출 수를 1/20으로 줄인다.
🟣 Sonnet — 요약 + 이슈 전담
플랫폼당 단 1회 호출. 전체 분류 통계 + 버그 리뷰 원문을 입력으로 받아 요약과 이슈 목록을 동시 반환.
왜 Sonnet인가
"버그 리뷰들에서 공통 이슈를 추출해 한국어로 요약"은 추론과 합성이 필요한 작업. 리뷰 수가 아무리 많아도 Sonnet은 1회만 호출하므로 비용이 고정된다.
Sonnet 1회로 요약 + 이슈 동시 생성
프롬프트에 버그 원문을 추가해 JSON으로 받는다
Haiku가 버그 카테고리로 분류한 리뷰의 원문을 최대 20건 수집해 Sonnet 프롬프트에 함께 전달. 출력을 plain text가 아닌 JSON으로 지정해 두 값을 한번에 파싱.
// Sonnet 입력
{
  stats: { total, positive, negative, ... },
  topKeywords: [...],
  bugContents: ["리뷰 원문1", "리뷰 원문2", ...]
}

// Sonnet 출력
{
  "summary": "3문장 한국어 요약",
  "issues": ["광고 스킵 불가", "결제 오류", ...]
}
왜 이 구조인가
이슈 목록을 별도 Sonnet 호출로 만들면 플랫폼당 2회, iOS+Android 합계 4회가 된다. 프롬프트를 확장해 JSON으로 받으면 동일 비용으로 두 가지 산출물이 나온다.

Centiment — Incremental Analysis
증분 분석
동작 방식
이미 분석된 리뷰는 건너뛰고 신규 리뷰만 처리한다
1. review_analysis에서 기존 review_id 목록 조회
2. reviews에서 전체 리뷰 조회
3. review_id가 없는 리뷰만 필터 = 미분석 리뷰
4. 미분석 리뷰만 Haiku 분류
5. 신규 분석 행 INSERT
6. 전체 분석 결과 집계 후 Sonnet으로 summary 재생성 → 전체 rows UPDATE
왜
100건 중 5건이 신규라면 5건만 Haiku로 처리한다. 나머지 95건을 다시 분류하는 것은 같은 결과를 위한 낭비다.
review_id FK의 역할
어느 리뷰가 분석됐는지 추적하는 연결고리
review_analysis.review_id가 reviews.id를 참조. 이 FK 덕분에 "이 리뷰는 이미 분석됐다"를 DB 쿼리 한 번으로 파악할 수 있다.
-- 미분석 리뷰만 조회
SELECT * FROM reviews
WHERE id NOT IN (
  SELECT review_id
  FROM review_analysis
  WHERE review_id IS NOT NULL
)
전체 재분석(force) 모드
프롬프트를 바꾸거나 분류 기준이 변경됐을 때 사용. 기존 분석 결과를 모두 삭제하고 처음부터 재처리. 결과 페이지 헤더의 "전체 재분석" 버튼으로 접근.
리뷰 수집 레벨의 중복 처리
DB의 UNIQUE 제약이 중복 리뷰를 자동으로 막는다
App Store·Play Store에서 리뷰를 가져올 때 upsert + ignoreDuplicates: true로 insert. UNIQUE(app_id, platform, content, review_date, lang) 제약에 걸리는 행은 insert되지 않고 스킵. 신규 리뷰만 DB에 쌓이고, 증분 분석이 그 신규 리뷰만 처리하는 구조가 맞물린다.

Centiment — Cross-Game + Summary
게임 간 비교 + 핵심 결정 정리
게임 간 비교 — /cross
각 게임의 iOS+Android를 합산해 비교한다
Pizza Ready의 iOS 분석 + Android 분석을 하나로 집계해 "Pizza Ready 전체" 수치를 만든 뒤 Burger Please와 비교. 감성 비율, 카테고리 분포, 상위 키워드를 나란히 표시.
왜 플랫폼을 합산하나
타이틀 비교의 질문은 "Pizza Ready가 Burger Please보다 유저 만족도가 높은가"이다. 이 질문에 iOS vs Android 차이는 노이즈. 게임 전체의 대표 수치로 비교하는 것이 공정하다.
공통 키워드 강조
두 게임 모두에서 등장하는 키워드는 진한 배지로 표시. 공통 키워드는 특정 타이틀이 아닌 장르 전체의 특성을 나타낸다.
핵심 설계 결정 요약
결정 이유
iOS+Android 동시 분석 비교 없는 단일 플랫폼 수치는 맥락이 없다
?game= 파라미터 iOS/Android가 각기 다른 app_id를 사용. 사용자 의미 단위는 게임
Haiku=분류, Sonnet=요약+이슈 분류는 패턴인식(저비용), 요약은 추론 필요(고품질). 역할에 맞는 모델
Sonnet 1회로 JSON 출력 추가 호출 없이 summary와 issues를 동시에 얻는다
20건 배치 처리 단건 대비 Haiku 호출 수 1/20. max_tokens 내 안정적 처리 가능한 크기
review_id FK + 증분 분석 신규 리뷰만 처리. 전체 재분류는 force 모드로 명시적 선택
키워드 드릴다운 로컬 state 일시적 탐색 행위. URL에 올릴 필요 없음
게임 간 비교 시 iOS+Android 합산 타이틀 비교에서 플랫폼 차이는 노이즈
7 / 7

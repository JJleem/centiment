왜 단방향인가
수집·분석·표시를 파이프라인으로 고정하면 각 단계를 독립적으로 교체하고 테스트할 수 있다. AI 분석 결과가 수집 로직에 영향을 주지 않기 때문에 어느 단계에서 문제가 생겼는지 즉시 격리할 수 있다.
DB 핵심 테이블
reviews
리뷰 원문 저장. UNIQUE(app_id, platform, content, review_date, lang) 제약으로 DB 레벨 중복 방지. lang 컬럼으로 로케일 추적.
왜 lang을 reviews에 저장하나
수집 시점에만 로케일 정보를 알 수 있다. 나중에 국가별 감성 분석을 위해 review_analysis.lang 컬럼에 전파한다.
review_analysis
review_id UUID → reviews.id FK로 어느 리뷰가 분석됐는지 추적. 증분 분석의 핵심 구조. lang 컬럼으로 국가별 감성 시각화 지원.
왜 FK가 필요한가
분석 완료된 리뷰 ID 집합을 O(1)로 조회해 미분석 리뷰만 골라낼 수 있다. content 기반 매칭보다 정확하고 빠르다.
cross_comparison_history
게임 간 비교 AI 인사이트 영구 저장. 동일 조합 재방문 시 Sonnet 재호출 없이 DB에서 즉시 반환. 재생성 버튼으로 수동 갱신 가능.
category_insights
카테고리별 Haiku 인사이트 캐싱. app_id·platform 단위로 저장. 첫 생성 이후엔 DB에서 즉시 반환, 새로고침 버튼으로 수동 재생성.

2컬럼 레이아웃 — 게임 그리드 + 스티키 패널
좌측: 게임 그리드 / 우측: 분석·비교 패널 (sticky)
게임 카드 클릭 시 우측 패널에 게임 정보·수집 건수 선택·분석 버튼이 즉시 표시. 스크롤 없이 모든 액션이 가능하다. 분석 완료된 게임엔 초록 도트 표시.
왜 스티키 패널인가
게임이 26개라 그리드가 길어진다. 게임 선택 후 액션 패널을 찾아 스크롤하는 UX를 없애기 위해 우측 패널을 화면에 고정했다.
주간 긍정률 요약 카드
최근 7일 분석 기준 TOP 3 / BOTTOM 3 자동 표시
게임 그리드 상단에 이번 주 긍정률 상위 3개(초록)와 하위 3개(빨강)를 자동으로 집계해 표시. 10건 미만 데이터는 통계 신뢰도 부족으로 제외.
왜
매주 모든 게임을 하나씩 확인하는 것은 비효율적이다. 이번 주에 주목해야 할 게임을 홈에서 즉시 파악할 수 있어야 한다.
미분석 게임 일괄 분석
한 번도 분석되지 않은 게임만 골라 체크박스 선택 후 순차 실행
선택한 게임들의 예상 API 비용($ 단위)을 미리 표시하고 확인 모달 후 실행. 게임당 수집 → 분석 순서로 처리하며 게임별 진행 상태(대기·수집중·분석중·완료·오류)를 실시간 표시.
왜 순차 처리인가
병렬로 실행하면 Anthropic API rate limit에 걸릴 수 있다. 한 게임씩 순서대로 처리해 안정적으로 완료를 보장한다.
게임 간 비교 진입 + 히스토리
분석된 게임 2~3개 선택 → /cross 이동 / 비교 히스토리 하단 표시
우측 패널 하단에 게임 1·2·3 드롭다운이 항상 표시. 과거 비교 기록(최근 10건)을 클릭하면 해당 비교 페이지로 즉시 이동. 분석 히스토리와 비교 히스토리가 나란히 배치.

왜 스트리밍인가
Haiku 배치 20건을 여러 번 호출하므로 전체 완료까지 수십 초가 걸린다. NDJSON 스트림으로 "iOS 분류 중 (3/8)" 같은 실시간 진행률을 보여줘 대기 불안을 줄인다.

Centiment — Result Page Features
결과 페이지 핵심 기능
버전 릴리즈 알림
연속 버전 간 긍정률 차이 ±10%p 이상 시 자동 배지
v1.2 → v1.3에서 긍정률이 20%p 하락하면 "급락 주의" 빨간 배지, 15%p 상승하면 "호반응" 초록 배지. iOS·Android 각각 독립 감지. 서버에서 versionTrend 데이터로 계산하므로 추가 API 호출 없음.
왜 10%p를 임계값으로 쓰나
단순 샘플 변동은 5%p 이내에서 발생한다. 10%p 이상이면 업데이트 내용이 유저 반응에 실질적 영향을 준 것으로 판단할 수 있다.
셀링포인트 추출
긍정 리뷰 키워드만 별도 집계 — AI 호출 없음
전체 키워드 집계와 별개로, sentiment="positive"인 리뷰의 키워드만 따로 카운팅해 TOP 10을 표시. 순위별 색상 구분(#1 진초록, #2~3 연초록). 마케팅 카피 소재로 즉시 활용 가능.
왜 AI 호출 없이 가능한가
키워드는 이미 Haiku 분류 단계에서 각 리뷰별로 추출되어 DB에 저장되어 있다. 긍정 리뷰 필터링 + 빈도 집계만으로 충분하다.
카테고리별 세부 인사이트 (Haiku)
상위 3개 카테고리에 대해 Haiku가 한 줄 한국어 인사이트 생성
리뷰를 카테고리별로 그룹핑 후 건수 상위 3개(bug·monetization·gameplay 등)를 선정. 각 카테고리의 샘플 리뷰를 Haiku에 전달해 "현재 유저가 이 카테고리에서 무엇을 느끼는지" 한 줄로 요약. 결과는 category_insights 테이블에 저장해 재방문 시 재호출 없이 즉시 반환.
왜 Haiku인가
카테고리 인사이트는 짧고 단순한 패턴 인식이라 Haiku로 충분하다. Sonnet 대비 비용이 약 10분의 1이다.
국가별 감성 분포
EN·KO·JA·DE 로케일별 긍정/부정/중립 비율 시각화
분석 파이프라인에서 reviews.lang을 review_analysis.lang으로 전파. 국가별로 긍정률이 얼마나 다른지 국기 이모지 + 감성 바로 표시. 재분석(force mode) 후 즉시 데이터 확보.
왜 DB 스키마 변경이 필요했나
기존 review_analysis에 lang 컬럼이 없었다. FK가 없는 인덱스 기반 페어링 구조상 lang을 나중에 JOIN할 수 없어 분석 시점에 함께 저장하는 구조로 변경했다.
CSV 내보내기 + 리뷰 원문 검색
CSV 내보내기 — 날짜·플랫폼·감성·카테고리·키워드·본문 포함
클라이언트에서 Blob 생성 후 즉시 다운로드. BOM(\uFEFF) 포함으로 Excel 한글 깨짐 방지. 파일명: 게임명_리뷰_YYYY-MM-DD.csv
리뷰 원문 검색 — 실시간 content 포함 여부 필터링
기존 플랫폼·감성·카테고리 필터와 AND 조건으로 복합 필터링. 검색어 매칭 텍스트를 노란색으로 하이라이트. X 버튼으로 즉시 초기화.

Centiment — Cross Comparison /cross
게임 간 비교 — /cross?g1=…&g2=…&g3=…
최대 3개 게임 동시 비교
g1·g2 필수, g3 선택. 중복 게임은 URL에서 자동 제거
2개 비교 시 2열 레이아웃, 3개 비교 시 3열 레이아웃으로 자동 전환. 감성 비교·카테고리 바·키워드 배지가 모두 N개 게임 기준으로 동적 생성. 3개 게임 공통 키워드는 진한 배지로 강조.
왜 3개까지 허용하나
동일 장르의 게임 3개를 한 화면에서 비교하면 "우리 게임만의 문제인가, 장르 전체의 문제인가"를 즉시 판단할 수 있다.
AI 인사이트 — Sonnet 한 문장 비교 요약
두 게임의 감성·카테고리·키워드 통계를 Sonnet에 전달해 한 줄 인사이트 생성
2개 비교: DB에서 기존 인사이트 조회 → 없으면 생성 후 저장 (영구 캐싱). 3개 비교: 항상 새로 생성 (히스토리 미저장). 재분석 버튼으로 수동 갱신 가능. "이미 라이브된 게임" 컨텍스트를 프롬프트에 명시해 "출시 전 전략" 환각 방지.
왜 DB 캐싱인가
같은 두 게임을 여러 팀원이 반복 조회할 수 있다. Sonnet 재호출 없이 즉시 반환하면 비용을 절약하고 응답 속도도 개선된다.
비교 히스토리
최근 10건 비교 기록을 홈 하단에 표시. 클릭 시 해당 비교 페이지 이동
2개 비교 시 cross_comparison_history 테이블에 게임명·인사이트·생성일 저장. 홈에서 CrossHistorySection이 클라이언트 사이드 fetch로 목록 표시. 이전에 어떤 게임을 비교했는지 이력 추적 가능.
색상 팔레트 — 게임별 고정 색상
게임 1: indigo / 게임 2: violet / 게임 3: fuchsia
감성 바·카테고리 바·키워드 배지·범례가 모두 동일한 색상 팔레트를 사용. 사용자가 페이지 내 어디서든 어떤 색이 어떤 게임인지 혼동 없이 파악할 수 있다.
왜 고정 팔레트인가
색상이 컨텍스트에 따라 달라지면 정보 처리 비용이 올라간다. 비교 페이지 전체에서 동일한 색=동일한 게임 규칙을 지켜야 빠르게 읽힌다.

Haiku 4.5 — 분류

sentiment: "positive" | "negative" | "neutral" category: "gameplay" | "ui" | "performance" | "monetization" | "content" | "bug" | "other" keywords: [최대 3개, 항상 영어, 소문자]

전체 기반 Sonnet 요약기존 + 신규 합산
왜 Sonnet 요약은 신규만이 아닌 전체 기반인가
요약은 지금까지 쌓인 모든 리뷰의 전체 맥락을 반영해야 한다. 신규 리뷰만 보면 최근 부정이 과대 반영되거나 장기 트렌드를 놓친다.
프롬프트 관리 원칙
lib/prompts/analyze.ts 에만 작성, 인라인 금지
buildClassifyPrompt · buildSummaryPrompt · buildCategoryInsightPrompt · buildCrossComparePrompt 4개 함수로 분리. 프롬프트 수정 시 한 파일만 변경하면 전체 반영.
